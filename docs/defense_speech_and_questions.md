# AuthGuard Locker қорғау мәтіні және комиссия сұрақтары

Бұл мәтін презентациядағы 20 слайд бойынша ауызша айтуға дайындалды. Слайдта аз жазылған жерлерді төмендегі сөздермен ашып айтуға болады. Екеуің сөйлегенде бір-біріңді қайталамай, Жанагүл көбіне кіріспе, интерфейс, қолданушы сценарийлерін айтса, Әсет архитектура, backend, қауіпсіздік және тестілеу жағын айтқаны дұрыс.

## Қысқа рөл бөлу

Жанагүл:
- 1-3 слайд: тақырып, өзектілік, мақсат-міндеттер.
- 6-12 слайд: қолданушы интерфейсі, тіркелу, dashboard, құжат жүктеу, бөлісу.
- 17, 20 слайд: қолдану салалары және аяқтау сөзі.

Әсет:
- 4-5 слайд: архитектура, ER-диаграмма, жүйе артықшылықтары.
- 9, 13-16 слайд: 2FA, криптографиялық модуль, admin panel, салыстыру.
- 18-19 слайд: тестілеу нәтижелері, қорытынды.

## 1-слайд. Тақырып

Жанагүл:
Қайырлы күн, құрметті комиссия мүшелері! Біздің дипломдық жобамыздың тақырыбы: "Қазіргі аутентификация және шифрлау әдістерін пайдалана отырып қорғалған веб-қосымша әзірлеу". Жобаның атауы - AuthGuard Locker. Бұл жүйе пайдаланушылардың аккаунттарын қорғауға, құжаттарды қауіпсіз сақтауға және оларды бақыланатын түрде бөлісуге арналған.

Әсет:
Жоба 6В06301 - Ақпараттық қауіпсіздік білім беру бағдарламасы аясында орындалды. Біз бұл жұмыста веб-қосымша қауіпсіздігінің негізгі механизмдерін практикалық түрде іске асырдық: email арқылы тіркелу, JWT негізіндегі авторизация, екі факторлы аутентификация, AES-256-GCM арқылы файлдарды шифрлау, рөлдік басқару және әрекеттер журналын жүргізу.

## 2-слайд. Өзектілігі

Жанагүл:
Бүгінгі таңда ұйымдар құжаттарды цифрлық түрде сақтап, онлайн бөліседі. Бірақ құжат айналымы артқан сайын жеке деректердің таралу қаупі де өседі. Қазақстанда да кибершабуылдар саны көбейіп жатыр. Сондықтан жай ғана файл сақтау жүйесі емес, аутентификациясы, авторизациясы және шифрлауы бар қорғалған веб-қосымша қажет.

Әсет:
Осы өзектілікке байланысты біз жобада бірнеше қауіпсіздік қабатын қарастырдық. Бірінші қабат - пайдаланушыны дұрыс анықтау, яғни email, пароль және қажет болса 2FA. Екінші қабат - API деңгейіндегі JWT токені арқылы қорғалған маршруттар. Үшінші қабат - құжаттарды серверде ашық түрде емес, шифрланған күйде сақтау. Төртінші қабат - әрекеттерді журналға жазу.

## 3-слайд. Мақсаты мен міндеттері

Жанагүл:
Жобаның негізгі мақсаты - құжаттарды сақтау және оларға қолжетімділікті басқару кезінде ақпараттық қауіпсіздік деңгейін арттыратын веб-қосымша әзірлеу. Яғни қолданушы файлды жүктегенде, ол файл қорғалып сақталуы керек, ал басқа адамға жібергенде немесе уақытша сілтеме бергенде қолжетімділік бақылануы керек.

Әсет:
Міндеттер бірнеше бағытқа бөлінді. Біріншісі - қауіпсіз аутентификация жасау. Екіншісі - TOTP негізіндегі 2FA енгізу. Үшіншісі - құжаттарды AES-256-GCM алгоритмі арқылы шифрлау. Төртіншісі - құжаттарды жүктеу, көру, жіберу және уақытша сілтеме арқылы бөлісу функцияларын жасау. Бесіншісі - admin panel және әрекеттер тарихын қосып, жүйені тестілеу.

## 4-слайд. Архитектура және ER-диаграмма

Әсет:
Жүйе клиент-сервер архитектурасы бойынша құрылған. Frontend бөлігі React және Vite арқылы жасалды. Ол қолданушы интерфейсін көрсетеді және backend API-ға сұраныс жібереді. Backend бөлігі Node.js және Express.js негізінде жасалды. Ол тіркелу, кіру, 2FA, құжаттарды өңдеу, шифрлау, рөлдерді тексеру сияқты негізгі логиканы орындайды.

Жанагүл:
Деректер базасында users, documents, shared_links, activity_logs, refresh_tokens сияқты кестелер бар. Users кестесінде қолданушы деректері, пароль хэші, рөлі және 2FA параметрлері сақталады. Documents кестесінде құжат туралы ақпарат және шифрланған файл дерегі сақталады. Shared_links уақытша сілтемелерді, ал activity_logs жүйедегі маңызды әрекеттерді тіркейді.

## 5-слайд. Жүйенің артықшылықтары

Әсет:
Жүйенің бірінші артықшылығы - көпдеңгейлі қауіпсіздік. Парольдер ашық түрде сақталмайды, bcrypt арқылы хэштеледі. Қорғалған API-ларға тек JWT токені арқылы кіруге болады. Қосымша қорғаныс ретінде 2FA бар.

Жанагүл:
Екінші артықшылығы - құжаттардың құпиялылығы. Пайдаланушы файл жүктеген кезде файл серверде шифрланған түрде сақталады. Үшінші артықшылық - қауіпсіз бөлісу. Құжатты басқа тіркелген қолданушыға жіберуге немесе уақытша сілтеме арқылы бөлісуге болады. Төртінші артықшылық - admin panel арқылы қолданушыларды басқару және әрекеттерді бақылау.

## 6-слайд. AuthGuard Locker басты беті

Жанагүл:
Бұл слайдта жүйенің бастапқы беті көрсетілген. Басты бет пайдаланушыға жүйенің мақсатын түсіндіреді және тіркелу немесе кіру әрекеттеріне бағыттайды. Біз интерфейсті қарапайым әрі түсінікті жасауға тырыстық, себебі қауіпсіздік жүйесі күрделі болғанымен, қолданушыға ыңғайлы болуы керек.

Әсет:
Frontend-та компоненттік тәсіл қолданылды. Әр бет жеке React компонент ретінде жасалды. Бұл кодты басқаруды жеңілдетеді және кейін жаңа функция қосқанда жүйені бұзбай кеңейтуге мүмкіндік береді.

## 7-слайд. Тіркелу және кіру беті

Жанагүл:
Тіркелу кезінде қолданушы email енгізеді, жүйе растау кодын email-ға жібереді. Код дұрыс болса ғана келесі қадамға өтеді. Содан кейін аты-жөні мен құпия сөзін енгізіп аккаунт жасайды. Бұл жерде әр қолданушы парольді өзі енгізуі керек, браузердің автоматты толтыруын шектеу үшін form autocomplete параметрлері қолданылды.

Әсет:
Backend-та тіркелу кезінде пароль bcryptjs арқылы хэштеледі. Кіру кезінде енгізілген пароль хэшпен салыстырылады. Егер дұрыс болса, сервер JWT access token және refresh token береді. Егер қолданушыда 2FA қосылған болса, толық токен бірден берілмейді, алдымен уақытша 2FA token беріледі, содан кейін TOTP коды тексеріледі.

## 8-слайд. Dashboard және профиль

Жанагүл:
Dashboard - қолданушының негізгі жұмыс аймағы. Бұл жерден құжаттарға, профильге, қауіпсіздік баптауларына, әрекеттер тарихына және криптографиялық модульге өтуге болады. Профиль бетінде қолданушы өзінің ақпаратын көре алады.

Әсет:
Frontend сұраныстары Axios арқылы backend-ке жіберіледі. JWT токені Authorization header арқылы қосылады. Егер токен жарамсыз болса немесе мерзімі бітсе, қорғалған маршруттарға кіруге рұқсат берілмейді.

## 9-слайд. 2FA баптау беті

Әсет:
2FA TOTP технологиясы арқылы іске асырылды. TOTP - уақытқа негізделген бір реттік код. Сервер speakeasy кітапханасы арқылы әр қолданушыға құпия кілт жасайды, ал qrcode кітапханасы QR-код генерациялайды. Қолданушы оны Google Authenticator сияқты қосымша арқылы сканерлейді.

Жанагүл:
2FA қосылғаннан кейін жүйеге кіру екі кезеңмен жүреді. Бірінші кезеңде email және пароль тексеріледі. Екінші кезеңде authenticator қосымшасындағы 6 таңбалы код енгізіледі. Бұл пароль белгілі болып қалған жағдайда да аккаунтты қорғауға көмектеседі.

## 10-слайд. Құжат жүктеу және қарау

Жанагүл:
Бұл бетте қолданушы құжаттарды жүктей алады және кейін оларды қауіпсіз түрде қарай алады. Жүйеде PDF, Word, Excel, PowerPoint, сурет және мәтіндік файлдармен жұмыс істеу қарастырылған. Құжаттар "Менің құжаттарым" бөлімінде сақталады.

Әсет:
Файл жүктелген кезде backend оны қабылдайды, файл типі мен көлемін тексереді, содан кейін AES-256-GCM арқылы шифрлайды. AES-256-GCM таңдалу себебі - ол тек шифрлап қана қоймай, деректің өзгермегенін де тексеруге мүмкіндік береді. GCM режимінде authentication tag болады, егер файл бұзылса немесе өзгертілсе, дешифрлау сәтсіз болады.

## 11-слайд. Уақытша сілтемемен бөлісу

Жанагүл:
Жүйеде құжатты уақытша сілтеме арқылы бөлісу мүмкіндігі бар. Қолданушы сілтеменің мерзімін таңдайды. Мерзімі өткеннен кейін сілтеме жарамсыз болады. Бұл құжатты тұрақты ашық қылып қоймай, уақыт бойынша шектеуге мүмкіндік береді.

Әсет:
Backend-та shared_links кестесінде token, document_id, expires_at және created_by сияқты өрістер сақталады. Сілтеме ашылған кезде сервер алдымен token-ді тексереді, содан кейін expires_at мерзімін салыстырады. Егер мерзімі бітсе, 410 жауап қайтарады.

## 12-слайд. Менің құжаттарым және маған келген құжаттар

Жанагүл:
Бұл бөлімде қолданушы өзінің жүктеген құжаттарын және басқа қолданушы жіберген құжаттарды көре алады. Құжаттарды іздеу, категория бойынша бөлу, жүктеп алу, қарау және өшіру мүмкіндіктері бар. Сондай-ақ құжатты басқа тіркелген қолданушыға email арқылы жіберу сценарийі қарастырылған.

Әсет:
Ресурсқа қолжетімділік backend-та тексеріледі. Яғни frontend-та батырма көрінбесе де, қауіпсіздік тек клиентке сеніп қалмайды. Әр сұраныста сервер қолданушының сол құжатқа құқығы бар-жоғын тексереді. Бұл IDOR типті осалдықтардың алдын алады.

## 13-слайд. Криптографиялық модуль

Әсет:
Криптографиялық модуль оқу және демонстрация мақсатында қосылды. Бұл жерде AES шифрлау, дешифрлау, SHA-256 хэштелу және RSA кілттерімен жұмыс сияқты процестерді тәжірибелік түрде көрсетуге болады. Бұл модуль негізгі құжат сақтау функциясынан бөлек, криптография принциптерін түсіндіруге арналған.

Жанагүл:
Комиссияға көрсеткенде бұл бет арқылы "шифрлау деген не, хэштеу деген не, симметриялық және асимметриялық криптография айырмашылығы қандай" деген сұрақтарға визуалды жауап беруге болады.

## 14-слайд. Әкімші панелі

Жанагүл:
Admin panel арқылы әкімші қолданушыларды көре алады, жаңа қолданушы қоса алады, рөлін өзгерте алады, 2FA баптауын reset жасай алады және қажет болса қолданушыны өшіре алады. Бұл ұйым ішінде қолжетімділікті орталықтан басқаруға мүмкіндік береді.

Әсет:
Рөлдік басқару backend-та жүзеге асады. Қолданушының role өрісі user немесе admin болуы мүмкін. Admin маршруттарына кірген кезде сервер JWT ішіндегі қолданушыны тексеріп, деректер базасынан оның рөлін қарайды. Тек admin болса ғана басқару әрекеттеріне рұқсат беріледі.

## 15-слайд. Әрекет тарихы

Жанагүл:
Жүйеде маңызды әрекеттер журналға жазылады: тіркелу, кіру, шығу, құжат жүктеу, құжат қарау, бөлісу, өшіру, 2FA әрекеттері. Бұл әкімшіге және қолданушыға жүйедегі белсенділікті бақылауға мүмкіндік береді.

Әсет:
Activity logs қауіпсіздік тұрғысынан маңызды. Егер бір аккаунтта күмәнді әрекет болса, мысалы көп рет қате кіру немесе құжатты жиі жүктеу, оны журнал арқылы көруге болады. Логта action_type, action_details және created_at сақталады.

## 16-слайд. Қолданыстағы жүйелермен салыстыру

Әсет:
Бұл слайдта AuthGuard Locker Google Drive және Yandex Disk сияқты жүйелермен салыстырылады. Олар файл сақтау және бөлісу үшін өте ыңғайлы, бірақ біздің жобаның ерекшелігі - оқу жобасы ретінде қауіпсіздік механизмдерін толық көрсетуі: 2FA, әрекеттер тарихы, рөлдік басқару, криптографиялық модуль, уақытша сілтеме мерзімін орнату.

Жанагүл:
Біздің жүйе үлкен коммерциялық бұлттық сервиспен бәсекелесу үшін емес, ұйым ішіндегі құжаттарды қауіпсіз сақтауға және ақпараттық қауіпсіздік механизмдерін практикалық түрде көрсетуге бағытталған.

## 17-слайд. Қолдану мүмкіндіктері

Жанагүл:
Жүйені HR бөлімдерінде қызметкерлердің құжаттарын қауіпсіз сақтау үшін қолдануға болады. Компаниялар ішінде келісімшарттар, анықтамалар, ішкі құжаттар сияқты файлдарды рөлге байланысты басқаруға болады. Бөлім басшылары құжаттарды уақытша сілтеме арқылы жіберіп, мерзімін шектей алады.

Әсет:
Сондай-ақ жүйені білім беру ұйымдарында студенттердің құжаттарын, өтініштерін немесе ішкі материалдарды қорғау үшін бейімдеуге болады. Болашақта электрондық қолтаңба, толық аудит, cloud storage немесе корпоративтік LDAP интеграциясы қосылуы мүмкін.

## 18-слайд. Тестілеу нәтижелері

Әсет:
Тестілеу барысында негізгі функциялар тексерілді: тіркелу, email код, кіру, 2FA қосу, құжат жүктеу, шифрланған сақтау, құжатты қарау, уақытша сілтеме жасау, басқа қолданушыға жіберу, admin panel және әрекеттер тарихы. Қауіпсіздік бойынша да тексерулер жүргізілді: дұрыс емес токенмен кіру, басқа қолданушы құжатына қол жеткізу, қате пароль, мерзімі өткен сілтеме.

Жанагүл:
Нәтижесінде қорғалған веб-қосымша әзірленді. Пайдаланушы интерфейсі жұмыс істейді, backend API сұраныстарды өңдейді, деректер PostgreSQL базасында сақталады, ал құжаттар шифрланған түрде қорғалады.

## 19-слайд. Қорытынды

Жанагүл:
Дипломдық жоба барысында AuthGuard Locker қорғалған веб-қосымшасы әзірленді. Жүйе құжаттарды қауіпсіз сақтау, басқару және бөлісуге арналған. Пайдаланушыға ыңғайлы интерфейс жасалып, тіркелу, кіру, 2FA, dashboard, құжат жүктеу және бөлісу мүмкіндіктері іске асырылды.

Әсет:
Техникалық жағынан жобада React және Vite frontend, Node.js және Express.js backend, PostgreSQL деректер базасы, JWT авторизациясы, bcrypt пароль хэші, TOTP 2FA, AES-256-GCM файл шифрлауы қолданылды. Жоба ақпараттық қауіпсіздіктің бірнеше негізгі механизмін бір жүйеге біріктіреді.

## 20-слайд. Назар аударғаныңыз үшін рахмет

Жанагүл:
Құрметті комиссия мүшелері, назарларыңызға рахмет! Біз AuthGuard Locker жүйесін көрсетуге дайынбыз.

Әсет:
Сұрақтарыңыз болса, жауап беруге дайынбыз.

## Демо кезінде айтылатын қысқа сценарий

1. Басты беттен тіркелу формасын ашамыз.
2. Email енгізіп, растау кодын жібереміз.
3. Email-ға келген кодты енгізіп, парольді өзіміз жазамыз.
4. Жүйеге кіріп, dashboard бетін көрсетеміз.
5. 2FA баптау бетінде QR-код арқылы қосу принципін түсіндіреміз.
6. Құжат жүктейміз және оны "Менің құжаттарым" бөлімінде көрсетеміз.
7. Құжаттың қауіпсіз сақталатынын, backend-та AES-256-GCM қолданылатынын айтамыз.
8. Уақытша сілтеме жасап, мерзімі бар екенін көрсетеміз.
9. Admin panel-де қолданушы рөлдерін басқару мүмкіндігін көрсетеміз.
10. Әрекет тарихында логтар жазылатынын көрсетеміз.

## Комиссия сұрақтары және дайын жауаптар

1. Жобаңыздың негізгі мақсаты қандай?
Жауап: Құжаттарды қауіпсіз сақтау, басқару және бөлісуге арналған қорғалған веб-қосымша әзірлеу.

2. Неге бұл тақырып өзекті?
Жауап: Құжат айналымы цифрлық форматқа көшкен сайын жеке деректердің таралу қаупі өседі, сондықтан аутентификация, авторизация және шифрлау қажет.

3. Жүйенің атауы қандай?
Жауап: AuthGuard Locker.

4. Frontend қандай технологиямен жасалды?
Жауап: React және Vite арқылы жасалды.

5. Backend қандай технологиямен жасалды?
Жауап: Node.js және Express.js арқылы жасалды.

6. Деректер базасы қандай?
Жауап: PostgreSQL қолданылды, код SQL Server-ге де бейімделген.

7. Неге React таңдадыңыз?
Жауап: React компоненттік архитектура береді, интерфейсті бөліктерге бөліп жасауға ыңғайлы және SPA құруға қолайлы.

8. Неге Vite қолданылды?
Жауап: Vite development кезінде жылдам hot reload береді және production build-ті тез жасайды.

9. Неге Node.js таңдадыңыз?
Жауап: Node.js асинхронды I/O моделі арқылы API сұраныстарын тиімді өңдейді және frontend-пен бір тілде, JavaScript-та жұмыс істеуге мүмкіндік береді.

10. Express.js не үшін керек?
Жауап: REST API маршруттарын, middleware-ді және сұраныс-жауап логикасын жеңіл ұйымдастыру үшін.

11. JWT деген не?
Жауап: JWT - сервер қол қойған токен. Ол қолданушының кім екенін және қандай рөлі бар екенін API сұраныстарында тексеруге мүмкіндік береді.

12. JWT қандай бөліктерден тұрады?
Жауап: Header, Payload және Signature.

13. JWT қайда қолданылады?
Жауап: Қорғалған API маршруттарына кіргенде Authorization header арқылы жіберіледі.

14. Access token мен refresh token айырмашылығы қандай?
Жауап: Access token қысқа мерзімге API-ға кіруге арналған, refresh token жаңа access token алу үшін ұзақтау сақталады.

15. Парольдер қалай сақталады?
Жауап: Ашық түрде сақталмайды, bcryptjs арқылы хэштеледі.

16. Неге парольді шифрламай, хэштейміз?
Жауап: Хэш біржақты функция. Парольді қайта ашу мүмкін болмауы керек, тек енгізілген парольдің хэші салыстырылады.

17. bcrypt артықшылығы қандай?
Жауап: Ол баяу хэш функциясы, salt қолданады және brute force шабуылын қиындатады.

18. 2FA деген не?
Жауап: Екі факторлы аутентификация. Парольден бөлек қосымша бір реттік код талап етеді.

19. TOTP деген не?
Жауап: Time-based One-Time Password, уақытқа негізделген бір реттік код.

20. 2FA қалай іске асырылды?
Жауап: speakeasy арқылы құпия кілт жасалады, qrcode арқылы QR көрсетіледі, қолданушы authenticator app-пен код алады.

21. 2FA коды қанша таңбадан тұрады?
Жауап: Әдетте 6 таңбадан тұрады.

22. Егер қолданушы 2FA телефонын жоғалтса ше?
Жауап: Admin panel арқылы 2FA reset жасау функциясы қарастырылған.

23. AES-256-GCM не үшін қолданылды?
Жауап: Құжаттарды серверде шифрланған күйде сақтау үшін.

24. AES симметриялық па, асимметриялық па?
Жауап: Симметриялық алгоритм, бір құпия кілтпен шифрлау және дешифрлау жасалады.

25. GCM режимінің артықшылығы қандай?
Жауап: Ол деректі шифрлап қана қоймай, authentication tag арқылы тұтастығын тексереді.

26. Файл қашан шифрланады?
Жауап: Серверге жүктелгеннен кейін, сақталмай тұрып шифрланады.

27. Файл қашан дешифрланады?
Жауап: Қолданушы preview немесе download жасағанда, рұқсат тексерілгеннен кейін.

28. Құжатқа басқа қолданушы қол жеткізе ала ма?
Жауап: Тек иесі, жіберілген алушы немесе жарамды уақытша сілтемесі бар адам ғана.

29. IDOR шабуылынан қалай қорғандыңыз?
Жауап: Backend әр сұраныста құжат иесін немесе рұқсатты тексереді.

30. Уақытша сілтеме қалай жұмыс істейді?
Жауап: Сервер token жасайды, оны shared_links кестесінде сақтайды және expires_at арқылы мерзімін тексереді.

31. Сілтеме мерзімі өтсе не болады?
Жауап: Сервер құжатты көрсетпейді, expired жауап қайтарады.

32. Рөлдік басқару қалай жасалған?
Жауап: Users кестесінде role өрісі бар. Admin маршруттарына тек role=admin қолданушы кіре алады.

33. Admin panel не істейді?
Жауап: Қолданушыларды көру, қосу, өшіру, рөл өзгерту, 2FA reset жасау.

34. Activity log не үшін керек?
Жауап: Жүйедегі маңызды әрекеттерді бақылау және қауіпсіздік оқиғаларын талдау үшін.

35. Логта қандай деректер сақталады?
Жауап: action_type, action_details, created_at, кей жерде IP және user-agent.

36. Қолданушы әрекеттерін жазу құпиялылыққа әсер етпей ме?
Жауап: Логта құжат мазмұны сақталмайды, тек әрекет туралы қысқа мәлімет сақталады.

37. Email коды не үшін керек?
Жауап: Тіркеліп жатқан email шынымен қолданушыға тиесілі екенін растау үшін.

38. Email жіберу қандай сервис арқылы істейді?
Жауап: Mailjet API арқылы, қажет болса Brevo немесе Resend fallback ретінде қолдануға болады.

39. Email келмесе не болады?
Жауап: Жүйе кодты экранда көрсетпейді, email сервисін тексеру керек деген қате береді.

40. Неге кодты экранда көрсетпейсіз?
Жауап: Қауіпсіздік үшін. Код тек email иесіне баруы керек.

41. Frontend пен backend қалай байланысады?
Жауап: Axios арқылы REST API сұраныстары жіберіледі.

42. CORS не үшін керек?
Жауап: Frontend доменінен backend API-ға рұқсатты басқару үшін.

43. Helmet не үшін қолданылады?
Жауап: HTTP security headers қосу үшін.

44. Rate limiting не үшін қажет?
Жауап: Бір IP-ден тым көп сұранысты шектеу, brute force шабуылын азайту үшін.

45. Multer не үшін қолданылды?
Жауап: Файл жүктеу, multipart/form-data сұраныстарын өңдеу үшін.

46. Файл көлеміне шектеу бар ма?
Жауап: Иә, backend-та maximum upload size конфигурацияланады.

47. Қандай файл түрлері қолдайды?
Жауап: PDF, Word, Excel, PowerPoint, суреттер, мәтіндік файлдар.

48. PowerPoint preview қалай жасалады?
Жауап: Мүмкін болса LibreOffice арқылы PDF preview, болмаса PPTX ішіндегі мәтін HTML preview ретінде көрсетіледі.

49. Неге LibreOffice керек?
Жауап: Office файлдарын серверде PDF-ке конвертациялап, браузерде көрсету үшін.

50. Құжаттарды database-та сақтау дұрыс па?
Жауап: Кіші және орта файлдар үшін мүмкін. Үлкен production жүйеде object storage қолданған дұрыс.

51. Production үшін не жақсартар едіңіз?
Жауап: Object storage, HttpOnly cookie, толық CSP, backup, monitoring, audit export қосар едік.

52. LocalStorage-та JWT сақтау қауіпсіз бе?
Жауап: Ыңғайлы, бірақ XSS болса қауіп бар. Production-та HttpOnly cookie жақсырақ.

53. XSS-тен қалай қорғанасыз?
Жауап: React әдепкі escaping жасайды, қауіпті HTML қолданбауға тырыстық, input validation бар.

54. SQL injection қалай алдын алынды?
Жауап: Parameterized query және prepared input қолданылды.

55. CSRF қаупі бар ма?
Жауап: JWT Authorization header арқылы жіберілсе CSRF қаупі cookie-based auth-қа қарағанда төмен.

56. Құпия кілттер қайда сақталады?
Жауап: Environment variables ішінде, кодқа commit жасалмайды.

57. ENCRYPTION_KEY не үшін керек?
Жауап: AES-256-GCM шифрлау кілтін алу үшін.

58. JWT_SECRET не үшін керек?
Жауап: JWT токеніне қол қою және оны тексеру үшін.

59. PostgreSQL-де қандай негізгі кестелер бар?
Жауап: users, documents, shared_links, activity_logs, refresh_tokens, token_blacklist.

60. Users кестесінде не бар?
Жауап: email, password_hash, role, is_verified, verification_code, twofa_secret, twofa_enabled.

61. Documents кестесінде не бар?
Жауап: title, category, filename, mime_type, file_size, file_data, user_id, deleted_at.

62. Shared_links кестесінде не бар?
Жауап: document_id, token, expires_at, created_by.

63. Refresh token не үшін blacklist/revoke болады?
Жауап: Logout немесе token rotation кезінде ескі token қолданылмауы үшін.

64. Token blacklist не үшін керек?
Жауап: Access token мерзімі бітпей тұрып logout жасалғанда оны жарамсыз ету үшін.

65. Басты қауіпсіздік қағидаңыз қандай?
Жауап: Defense in depth, яғни бірнеше қорғаныс қабатын қолдану.

66. Жүйеде қандай қолданушы рөлдері бар?
Жауап: user және admin.

67. Admin өзін өшіре ала ма?
Жауап: Интерфейсте өзін өшіру немесе admin рөлін алып тастау шектелген.

68. Құжатты өшіргенде бірден жойыла ма?
Жауап: Жүйеде deleted_at арқылы soft delete логикасы бар, қажет болса қалпына келтіру сценарийі қолданылады.

69. Себет функциясы бар ма?
Жауап: Иә, өшірілген құжаттарды бөлек көрсету және басқару қарастырылған.

70. Құжатты басқа user-ге жіберу қалай жұмыс істейді?
Жауап: Алушы email бойынша users кестесінен табылады, document_transfers арқылы байланыс жасалады.

71. Алушы тіркелмеген болса ше?
Жауап: Жүйе "тіркелген қолданушы табылмады" деп қате қайтарады.

72. Неге тіркелмеген email-ге құжат жібермейсіз?
Жауап: Құжатқа қолжетімділікті жүйе ішінде бақылау үшін.

73. Уақытша сілтемені кез келген адам аша ала ма?
Жауап: Иә, token бар адам аша алады, бірақ мерзімі шектеулі.

74. Уақытша сілтеменің қауіпі қандай?
Жауап: Token басқа адамға түссе ашуы мүмкін, сондықтан мерзім шектеуі және күрделі random token қажет.

75. Token қалай жасалады?
Жауап: Crypto random мәндері арқылы болжанбайтын token жасалады.

76. Құжат preview қауіпсіз бе?
Жауап: Preview алдында рұқсат тексеріледі және файл тек сұраған кезде дешифрланады.

77. PDF пен DOCX preview айырмашылығы?
Жауап: PDF браузерде тікелей ашылады, DOCX/Office файлдары конвертация немесе HTML preview арқылы көрсетіледі.

78. Неге криптографиялық модуль қостыңыз?
Жауап: Шифрлау, хэштеу, RSA принциптерін тәжірибе жүзінде көрсету үшін.

79. SHA-256 не үшін керек?
Жауап: Деректен тұрақты хэш алу, тұтастықты тексеру және криптографияны түсіндіру үшін.

80. RSA не үшін көрсетілген?
Жауап: Асимметриялық криптография принципін көрсету үшін.

81. AES пен RSA айырмашылығы қандай?
Жауап: AES симметриялық, бір кілт; RSA асимметриялық, public/private key жұбы.

82. Жобаның ғылыми жаңалығы неде?
Жауап: Бір оқу жобасында аутентификация, 2FA, JWT, AES-GCM, secure sharing және audit log механизмдері біріктірілді.

83. Практикалық маңызы қандай?
Жауап: HR, компания ішкі құжат айналымы, білім беру ұйымдарында құжаттарды қауіпсіз сақтау үшін қолдануға болады.

84. Google Drive-тен айырмашылығы қандай?
Жауап: Біздің жүйе ұйым ішіндегі қауіпсіздік механизмдерін ашық көрсетеді: рөлдік басқару, 2FA reset, криптомодуль, audit log.

85. Жүйе коммерциялық дайын өнім бе?
Жауап: Бұл дипломдық практикалық прототип, production үшін қосымша backup, monitoring және storage жақсарту керек.

86. Қандай тесттер жүргізілді?
Жауап: Функционалдық тесттер: тіркелу, кіру, құжат жүктеу, бөлісу, 2FA, admin panel. Қауіпсіздік тесттері: қате token, рұқсатсыз құжат, мерзімі өткен link.

87. Қандай қателер кездесті?
Жауап: Deployment кезінде database URL, email service, LibreOffice preview сияқты мәселелер болды, олар конфигурация және fallback логикасы арқылы түзетілді.

88. Render не үшін қолданылды?
Жауап: Backend-ті public API ретінде орналастыру үшін.

89. Vercel не үшін қолданылды?
Жауап: Frontend React қосымшасын орналастыру үшін.

90. Backend URL frontend-ке қалай беріледі?
Жауап: VITE_API_BASE_URL environment variable арқылы.

91. DATABASE_URL не үшін керек?
Жауап: Backend PostgreSQL деректер базасына қосылуы үшін.

92. SSL database connection керек пе?
Жауап: Cloud PostgreSQL қолданғанда PGSSLMODE=require арқылы SSL қосылады.

93. Жүйеде localization бар ма?
Жауап: Иә, қазақша, орысша, ағылшынша тіл ауыстыру қарастырылған.

94. Неге интерфейс қазақша?
Жауап: Қорғау және мақсатты қолданушы аудиториясы үшін қазақша негізгі тіл ретінде таңдалды.

95. Неге email код 10 минут жарамды?
Жауап: Қауіпсіздік пен ыңғайлылық арасындағы баланс. Ұзақ болса қауіп, өте қысқа болса қолданушыға ыңғайсыз.

96. Пароль саясаты қандай?
Жауап: Кемінде 8 таңба, бас әріп, кіші әріп және сан талап етіледі.

97. Қолданушы парольді ұмытса ше?
Жауап: Forgot password арқылы email-ға reset code жіберіліп, жаңа пароль қойылады.

98. Email service істемесе не болады?
Жауап: Код экранда көрсетілмейді, жүйе email сервисін тексеру қажет екенін хабарлайды.

99. Жобаны болашақта қалай дамытасыз?
Жауап: HttpOnly cookie, object storage, CSP, full audit export, backup, notifications, электрондық қолтаңба қосуға болады.

100. Жобаның қорытынды нәтижесі қандай?
Жауап: AuthGuard Locker атты қорғалған веб-қосымша әзірленді, онда тіркелу, кіру, 2FA, құжаттарды шифрлау, қауіпсіз бөлісу, admin panel және әрекеттер журналы іске асырылды.

## Ең жиі қойылатын 10 сұраққа өте қысқа жауап

1. Неге AES-256-GCM? - Құпиялылық пен тұтастықты бірге береді.
2. Неге bcrypt? - Парольді ашық сақтамайды және brute force-ты қиындатады.
3. Неге JWT? - Stateless API authorization жасауға ыңғайлы.
4. Неге 2FA? - Пароль ұрланса да қосымша қорғаныс береді.
5. Неге React? - Компоненттік, жылдам, SPA жасауға ыңғайлы.
6. Неге Node/Express? - REST API тез құрылады, JavaScript ecosystem кең.
7. Неге PostgreSQL? - Сенімді relational database, cloud-та оңай deploy.
8. Құжат қалай қорғалады? - Рұқсат тексеріледі, файл AES-GCM арқылы шифрланады.
9. Admin не істейді? - Қолданушы, рөл, 2FA және жүйе күйін басқарады.
10. Болашақ жақсарту? - HttpOnly cookie, cloud object storage, backup, CSP, monitoring.
