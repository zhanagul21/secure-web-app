# AuthGuard Locker қорғау мәтіні және комиссия сұрақтары

Бұл мәтін презентациядағы 20 слайд бойынша ауызша айтуға дайындалды. Слайдта аз жазылған жерлерді төмендегі сөздермен ашып айтуға болады. Екеуің сөйлегенде бір-біріңді қайталамай, Жанагүл көбіне кіріспе, интерфейс, қолданушы сценарийлерін айтса, Әсет архитектура, backend, қауіпсіздік және тестілеу жағын айтқаны дұрыс.

## Қысқа рөл бөлу

Жанагүл:
- 1-3 слайд: тақырып, өзектілік, мақсат-міндеттер.
- 6-12 слайд: қолданушы интерфейсі, тіркелу, dashboard, құжат жүктеу, бөлісу.
- 17, 20 слайд: қолдану салалары және аяқтау сөзі.

Әсет:
- 4-5 слайд: архитектура, ER-диаграмма, жүйе артықшылықтары.
- 9, 13-16 слайд: 2FA, криптографиялық модуль, admin panel, салыстыру.
- 18-19 слайд: тестілеу нәтижелері, қорытынды.

## 1-слайд. Тақырып

Жанагүл:
Қайырлы күн, құрметті комиссия мүшелері! Біздің дипломдық жобамыздың тақырыбы: "Қазіргі аутентификация және шифрлау әдістерін пайдалана отырып қорғалған веб-қосымша әзірлеу". Жобаның атауы - AuthGuard Locker. Бұл жүйе пайдаланушылардың аккаунттарын қорғауға, құжаттарды қауіпсіз сақтауға және оларды бақыланатын түрде бөлісуге арналған.

Әсет:
Жоба 6В06301 - Ақпараттық қауіпсіздік білім беру бағдарламасы аясында орындалды. Біз бұл жұмыста веб-қосымша қауіпсіздігінің негізгі механизмдерін практикалық түрде іске асырдық: email арқылы тіркелу, JWT негізіндегі авторизация, екі факторлы аутентификация, AES-256-GCM арқылы файлдарды шифрлау, рөлдік басқару және әрекеттер журналын жүргізу.

## 2-слайд. Өзектілігі

Жанагүл:
Бұл слайдта біз жобаның өзектілігін нақты дереккөзге сүйеніп көрсетіп отырмыз. Times of Central Asia сайтындағы "Cyberattacks Double in Kazakhstan in Early 2025" атты мақалада 2025 жылдың қаңтар-наурыз айларында Қазақстанда ақпараттық қауіпсіздік бойынша шамамен 30 мың инцидент тіркелгені айтылған. Бұл 2024 жылдың осы кезеңімен салыстырғанда екі есе көп. Яғни кибершабуылдар саны азаймай, керісінше жыл сайын өсіп отыр.

Осы дерекке сүйенсек, қазіргі кезде тек сайт жасау жеткіліксіз. Әсіресе құжатпен жұмыс істейтін жүйелерде пайдаланушының жеке деректері, файлдары, ішкі құжаттары және құпия ақпараттары қорғалуы керек. Егер мұндай жүйеде дұрыс аутентификация болмаса, парольдер ашық сақталса немесе файлдар шифрланбаса, шабуылдаушы аккаунтқа кіріп, құжаттарды ұрлауы немесе таратып жіберуі мүмкін.

Әсет:
Сондықтан біздің жобамыздың өзектілігі дәл осы мәселеден шығады: құжаттарды қауіпсіз сақтау, қолданушыны сенімді түрде тексеру және қолжетімділікті басқару қажет. AuthGuard Locker осы талаптарды орындау үшін жасалды. Жүйеде email арқылы растау, парольді bcrypt арқылы хэштеу, JWT токенімен қорғалған API, TOTP негізіндегі екі факторлы аутентификация және AES-256-GCM арқылы файлдарды шифрлау қолданылды.

Басқаша айтқанда, слайдтағы дерек бізге мынаны көрсетеді: Қазақстанда ақпараттық қауіпсіздік инциденттері көбейіп жатқан жағдайда, қорғалған веб-қосымшаларды әзірлеу практикалық тұрғыдан маңызды. Біздің жүйе осы мәселені құжат сақтау саласында шешуге бағытталған.

Қысқа нұсқа, уақыт аз болса:
Times of Central Asia дерегіне сүйенсек, 2025 жылдың алғашқы тоқсанында Қазақстанда 30 мың ақпараттық қауіпсіздік инциденті тіркелген, бұл өткен жылмен салыстырғанда екі есе көп. Сондықтан құжаттарды ашық түрде сақтау қауіпті. Осы себепті біз пайдаланушыны email, пароль және 2FA арқылы тексеретін, API-ды JWT арқылы қорғайтын және файлдарды AES-256-GCM арқылы шифрлайтын AuthGuard Locker веб-қосымшасын әзірледік.

## 3-слайд. Мақсаты мен міндеттері

Жанагүл:
Жобаның негізгі мақсаты - құжаттарды сақтау және оларға қолжетімділікті басқару кезінде ақпараттық қауіпсіздік деңгейін арттыратын веб-қосымша әзірлеу. Яғни қолданушы файлды жүктегенде, ол файл қорғалып сақталуы керек, ал басқа адамға жібергенде немесе уақытша сілтеме бергенде қолжетімділік бақылануы керек.

Әсет:
Міндеттер бірнеше бағытқа бөлінді. Біріншісі - қауіпсіз аутентификация жасау. Екіншісі - TOTP негізіндегі 2FA енгізу. Үшіншісі - құжаттарды AES-256-GCM алгоритмі арқылы шифрлау. Төртіншісі - құжаттарды жүктеу, көру, жіберу және уақытша сілтеме арқылы бөлісу функцияларын жасау. Бесіншісі - admin panel және әрекеттер тарихын қосып, жүйені тестілеу.

## 4-слайд. Архитектура және ER-диаграмма

Әсет:
Бұл слайдта жүйенің жалпы архитектурасы және деректер базасының байланыстары көрсетілген. Біздің жүйе классикалық клиент-сервер архитектурасы бойынша жасалды. Жоба негізінен JavaScript тілінде жазылды: frontend бөлігі React және Vite арқылы, ал backend бөлігі Node.js және Express.js арқылы іске асырылды. Клиент жағы қолданушыға беттерді, формаларды, батырмаларды, құжаттар тізімін және қауіпсіздік баптауларын көрсетеді. Бірақ негізгі қауіпсіздік логикасы frontend-та емес, backend-та орындалады.

Backend бөлігі Node.js және Express.js арқылы жасалды. Backend API сұраныстарды қабылдайды, қолданушыны тексереді, JWT токен береді, 2FA кодын тексереді, құжатты қабылдайды, файлды шифрлайды және деректер базасымен жұмыс істейді. Яғни frontend тек интерфейс болса, backend жүйенің негізгі логикасы мен қауіпсіздік қабаты болып есептеледі.

Жүйенің жұмыс ағымы былай жүреді: қолданушы браузер арқылы сұраныс жібереді, frontend оны API-ға бағыттайды, backend сұранысты тексеріп, PostgreSQL деректер базасына жазады немесе сол жерден мәлімет алады. Егер сұраныс құжатқа қатысты болса, сервер алдымен рұқсатты тексереді, содан кейін ғана файлды көрсетеді немесе жүктеуге береді.

Жанагүл:
ER-диаграммада негізгі кестелердің байланысы көрсетілген. Бірінші негізгі кесте - users. Бұл кестеде қолданушының аты-жөні, email адресі, парольдің хэші, рөлі және 2FA параметрлері сақталады. Пароль ашық түрде сақталмайды, тек bcrypt арқылы алынған хэш сақталады.

Екінші маңызды кесте - documents. Мұнда құжаттың атауы, типі, көлемі, кім жүктегені және шифрланған файл туралы ақпарат сақталады. Әр құжат белгілі бір қолданушыға тиесілі болады, сондықтан documents кестесі users кестесімен байланысады.

Shared_links кестесі уақытша сілтемелер үшін керек. Онда сілтеме token-і, қай құжатқа қатысты екені және мерзімі сақталады. Activity_logs кестесі жүйедегі маңызды әрекеттерді тіркейді: тіркелу, кіру, құжат жүктеу, құжатты ашу, бөлісу және әкімші әрекеттері. Refresh_tokens кестесі сессияны қауіпсіз басқаруға көмектеседі. Осы кестелер бірге жұмыс істеп, қолданушы, құжат, рұқсат және әрекет тарихын бір жүйеге біріктіреді.

## 5-слайд. Жүйенің артықшылықтары

Әсет:
Бұл слайдта біздің жүйенің негізгі артықшылықтары көрсетілген. Бірінші артықшылық - көпдеңгейлі қауіпсіздік. Көп жүйелерде тек email мен пароль жеткілікті болуы мүмкін, бірақ біздің жобада бірнеше қорғаныс қабаты бар. Бірінші қабат - email арқылы тіркелуді растау. Екінші қабат - парольді bcrypt арқылы хэштеу. Үшінші қабат - JWT арқылы қорғалған API маршруттары. Төртінші қабат - TOTP негізіндегі 2FA.

Яғни бір ғана механизмге сүйенбей, бірнеше қауіпсіздік әдісін біріктірдік. Мысалы, шабуылдаушы парольді біліп қойса да, 2FA қосылған аккаунтқа бірден кіре алмайды. Ал токен болмаса, backend қорғалған маршруттарға рұқсат бермейді.

Жанагүл:
Екінші артықшылық - құжаттардың құпиялылығы. Қолданушы файл жүктеген кезде ол серверде жай файл ретінде емес, шифрланған түрде сақталады. Бұл сервердегі файлға тікелей қол жеткізілген жағдайда да оның мазмұнын бірден оқуға мүмкіндік бермейді.

Үшінші артықшылық - құжатты қауіпсіз бөлісу. Қолданушы құжатты басқа тіркелген қолданушыға жібере алады немесе уақытша сілтеме жасай алады. Уақытша сілтеменің мерзімі болады, сондықтан құжат интернетте шексіз уақыт ашық тұрмайды.

Төртінші артықшылық - admin panel және әрекеттер тарихы. Әкімші қолданушыларды басқара алады, рөлдерді өзгерте алады, қажет болса 2FA-ны қалпына келтіреді. Ал activity log арқылы жүйеде кім, қашан, қандай әрекет жасағанын көруге болады.

## 6-слайд. AuthGuard Locker басты беті

Жанагүл:
Бұл слайдта AuthGuard Locker жүйесінің бастапқы беті көрсетілген. Басты бет қолданушыға жүйенің не үшін жасалғанын бірден түсіндіреді: бұл жай файл сақтау сайты емес, қорғалған құжат сақтау және бөлісу платформасы. Пайдаланушы осы жерден тіркелу немесе жүйеге кіру әрекетін бастайды.

Интерфейсті жобалағанда біз қарапайымдыққа мән бердік. Себебі қауіпсіздік механизмдері backend жағында күрделі болуы мүмкін, бірақ қолданушы үшін жүйе түсінікті және ыңғайлы болуы керек. Егер қолданушыға интерфейс қиын болса, ол қауіпсіз функцияларды қолданбай кетуі мүмкін. Сондықтан негізгі әрекеттер - тіркелу, кіру, құжатқа өту - бірден көрінетіндей жасалды.

Әсет:
Техникалық жағынан frontend JavaScript тіліндегі React компоненттері арқылы жасалды. Мысалы, басты бет, тіркелу беті, dashboard, құжаттар беті, 2FA баптауы және admin panel жеке компоненттер ретінде бөлінді. Бұл тәсіл кодты құрылымды етеді.

Vite қолданылғандықтан, әзірлеу кезінде жоба тез іске қосылады және өзгерістер бірден көрінеді. Бұл frontend интерфейсін бірнеше рет тексеріп, дизайн мен логиканы жылдам түзетуге мүмкіндік берді. Сонымен қатар production build жеңіл және Vercel-ге орналастыруға ыңғайлы болды.

## 7-слайд. Тіркелу және кіру беті

Жанагүл:
Бұл слайдта тіркелу және кіру процесі көрсетілген. Тіркелу үш қадамнан тұрады. Бірінші қадамда қолданушы email адресін енгізеді. Жүйе осы email-ға растау кодын жібереді. Екінші қадамда қолданушы email-ға келген 6 таңбалы кодты енгізеді. Код дұрыс болса ғана үшінші қадам ашылады.

Үшінші қадамда қолданушы аты-жөнін және құпия сөзін өзі енгізеді. Мұнда маңызды нәрсе - парольді жүйе автоматты түрде жазып бермейді, әр қолданушы өз паролін өзі ойлап табуы керек. Бұл қауіпсіздік пен жеке жауапкершілік үшін қажет. Браузердің бұрынғы парольді автоматты қоюын азайту үшін form autocomplete параметрлері қолданылды.

Әсет:
Backend жағында тіркелу кезінде email коды тексеріледі, содан кейін пароль bcryptjs кітапханасы арқылы хэштеледі. Хэштеу дегеніміз - парольді қайта ашуға келмейтін біржақты түрге айналдыру. Деректер базасында нақты пароль емес, тек хэш сақталады.

Кіру кезінде қолданушы email мен пароль енгізеді. Сервер енгізілген парольді қайта хэштеп, деректер базасындағы хэшпен салыстырады. Егер дұрыс болса, access token және refresh token беріледі. Access token қорғалған API-ларға кіру үшін қолданылады. Егер қолданушыда 2FA қосылған болса, сервер толық кіруге бірден рұқсат бермейді. Алдымен уақытша 2FA token береді, содан кейін authenticator қосымшасындағы TOTP коды тексеріледі.

## 8-слайд. Dashboard және профиль

Жанагүл:
Dashboard - қолданушы жүйеге кіргеннен кейін көретін негізгі жұмыс аймағы. Бұл беттен қолданушы өзінің құжаттарына, профиль бетіне, 2FA қауіпсіздік баптауларына, әрекеттер тарихына және криптографиялық модульге өте алады. Яғни dashboard жүйенің навигациялық орталығы сияқты жұмыс істейді.

Профиль бетінде қолданушы өзінің аты-жөнін, email адресін және аккаунтқа қатысты негізгі ақпаратты көре алады. Бұл бет қолданушының жеке кабинеті ретінде қарастырылды.

Әсет:
Техникалық жағынан dashboard-тағы мәліметтер backend-тен API арқылы алынады. Frontend Axios кітапханасын қолданады. Әр қорғалған сұранысқа JWT access token Authorization header арқылы қосылады. Backend осы токенді тексеріп, қолданушының кім екенін анықтайды.

Егер токен жарамсыз болса, мерзімі бітсе немесе қолданушы жүйеге кірмеген болса, қорғалған маршруттарға кіруге рұқсат берілмейді. Бұл жерде қауіпсіздік тек интерфейсте емес, backend деңгейінде де тексеріледі.

## 9-слайд. 2FA баптау беті

Әсет:
Бұл слайдта екі факторлы аутентификация, яғни 2FA көрсетілген. 2FA дегеніміз - жүйеге кіру үшін парольден бөлек тағы бір дәлел сұрау. Біздің жобада 2FA TOTP технологиясы арқылы жасалды. TOTP - Time-based One-Time Password, яғни уақытқа негізделген бір реттік код.

Сервер speakeasy кітапханасы арқылы әр қолданушыға жеке құпия кілт жасайды. Сол кілт QR-код түрінде көрсетіледі. QR-кодты qrcode кітапханасы генерациялайды. Қолданушы оны Google Authenticator немесе Microsoft Authenticator сияқты қосымша арқылы сканерлейді. Осыдан кейін қосымша әр 30 секунд сайын жаңа 6 таңбалы код шығарып тұрады.

Жанагүл:
2FA қосылғаннан кейін кіру процесі екі кезеңмен жүреді. Бірінші кезеңде email және пароль тексеріледі. Екінші кезеңде қолданушы authenticator қосымшасындағы 6 таңбалы кодты енгізеді. Егер код дұрыс болса ғана жүйеге толық кіреді.

Бұл механизм пароль белгілі болып қалған жағдайда да аккаунтты қорғауға көмектеседі. Мысалы, біреу қолданушының паролін біліп қойса, authenticator қосымшасындағы кодсыз жүйеге кіре алмайды. Сондықтан 2FA аккаунт қауіпсіздігін едәуір күшейтеді.

## 10-слайд. Құжат жүктеу және қарау

Жанагүл:
Бұл слайдта құжат жүктеу және қарау процесі көрсетілген. Қолданушы жүйеге кіргеннен кейін PDF, Word, Excel, PowerPoint, сурет немесе мәтіндік файлдарды жүктей алады. Жүктелген файлдар "Менің құжаттарым" бөлімінде көрсетіледі.

Қолданушы үшін бұл процесс қарапайым көрінеді: файл таңдайды, жүктейді, кейін қарайды немесе жүктеп алады. Бірақ backend жағында бірнеше қауіпсіздік тексеруі орындалады. Файлдың типі, көлемі және қолданушының авторизациясы тексеріледі.

Әсет:
Файл серверге келгеннен кейін AES-256-GCM алгоритмі арқылы шифрланады. AES - симметриялық шифрлау алгоритмі, яғни шифрлау және дешифрлау үшін бір құпия кілт қолданылады. 256 саны кілт ұзындығын білдіреді, бұл қазіргі кезде сенімді деңгей болып саналады.

GCM режимінің артықшылығы - ол тек файлды шифрлап қана қоймай, файлдың өзгермегенін де тексереді. Шифрланған дерекпен бірге authentication tag сақталады. Егер файл бұзылса немесе біреу өзгертуге тырысса, дешифрлау кезінде қате шығады. Осылайша жүйе құжаттың құпиялылығын да, тұтастығын да қорғайды.

## 11-слайд. Уақытша сілтемемен бөлісу

Жанагүл:
Бұл слайдта құжатты уақытша сілтеме арқылы бөлісу мүмкіндігі көрсетілген. Кейде құжатты басқа адамға тез көрсету керек болады, бірақ оны тұрақты түрде ашық қалдыру қауіпті. Сондықтан біздің жүйеде мерзімі бар сілтеме жасау функциясы қосылды.

Қолданушы құжатты таңдап, сілтеменің қанша уақыт жарамды болатынын белгілейді. Мерзімі өткеннен кейін сілтеме автоматты түрде жарамсыз болады. Бұл құжатқа қолжетімділікті уақыт бойынша шектеуге мүмкіндік береді.

Әсет:
Backend жағында уақытша сілтеме үшін кездейсоқ token жасалады. Бұл token shared_links кестесінде document_id, expires_at және created_by өрістерімен бірге сақталады. Сілтеме ашылған кезде сервер алдымен token бар ма, дұрыс па, соны тексереді. Содан кейін expires_at уақыты қазіргі уақытпен салыстырылады.

Егер сілтеме мерзімі өтіп кетсе, сервер құжатты көрсетпейді және expired жауап қайтарады. Егер token дұрыс және мерзімі әлі бітпесе, сервер құжатты дешифрлап, preview ретінде көрсетеді. Осы арқылы құжатты бөлісу қауіпсіз әрі бақыланатын болады.

## 12-слайд. Менің құжаттарым және маған келген құжаттар

Жанагүл:
Бұл бөлімде қолданушы өзінің жүктеген құжаттарын және басқа қолданушы жіберген құжаттарды көре алады. "Менің құжаттарым" бөлімі жеке файлдарды басқаруға арналған. Ал "Маған келген құжаттар" бөлімі басқа қолданушылар жіберген файлдарды бөлек көрсетеді.

Құжаттарды іздеу, категория бойынша бөлу, қарау, жүктеп алу және өшіру мүмкіндіктері қарастырылған. Сонымен қатар қолданушы құжатты басқа тіркелген қолданушыға email арқылы жібере алады. Бұл ұйым ішінде құжат алмасуды ыңғайлы етеді.

Әсет:
Қауіпсіздік тұрғысынан ең маңыздысы - құжатқа қолжетімділік backend-та тексеріледі. Яғни frontend-та бір батырманы жасыру жеткіліксіз, себебі шабуылдаушы API-ға тікелей сұраныс жіберуі мүмкін. Сондықтан әр сұраныста сервер қолданушының сол құжатқа құқығы бар-жоғын тексереді.

Бұл IDOR деп аталатын осалдықтың алдын алады. IDOR - қолданушы URL немесе id параметрін өзгертіп, басқа адамның дерегіне қол жеткізуге тырысатын жағдай. Біздің жүйеде құжаттың иесі, алушысы немесе жарамды сілтемесі болмаса, backend құжатты көрсетпейді.

## 13-слайд. Криптографиялық модуль

Әсет:
Криптографиялық модуль оқу және демонстрация мақсатында қосылды. Бұл бет негізгі құжат сақтау функциясынан бөлек жұмыс істейді, бірақ жобаның тақырыбын жақсы ашады. Мұнда AES шифрлау және дешифрлау, SHA-256 арқылы хэштеу, RSA кілттерімен жұмыс сияқты процестерді көрсетуге болады.

AES симметриялық шифрлауды түсіндіреді: бір кілтпен шифрлау және сол кілтпен қайта ашу. SHA-256 хэштеу біржақты процесс екенін көрсетеді: мәтіннен хэш алуға болады, бірақ хэштен бастапқы мәтінді қайта шығару мүмкін емес. RSA асимметриялық криптографияны түсіндіреді: ашық және жабық кілттер жұбы қолданылады.

Жанагүл:
Комиссияға көрсеткен кезде бұл модуль арқылы теорияны практикамен байланыстырып түсіндіруге болады. Мысалы, "шифрлау деген не?", "хэштеу не үшін керек?", "симметриялық және асимметриялық криптографияның айырмашылығы қандай?" деген сұрақтарға осы бет арқылы көрнекі жауап береміз.

Бұл модуль жобаның білім беру құндылығын арттырады, себебі қолданушы қауіпсіздік механизмдерінің тек атауын ғана емес, олардың қалай жұмыс істейтінін де көре алады.

## 14-слайд. Әкімші панелі

Жанагүл:
Бұл слайдта әкімші панелі көрсетілген. Admin panel жүйені басқаруға арналған. Әкімші барлық қолданушыларды көре алады, жаңа қолданушы қоса алады, рөлін өзгерте алады, қажет болса қолданушыны өшіре алады және 2FA баптауын reset жасай алады.

Мұндай мүмкіндік ұйым ішінде маңызды, себебі барлық қолданушыға бірдей құқық беруге болмайды. Кейбір қолданушылар тек құжат жүктеп, қарайды, ал admin жүйені басқарады. Сондықтан рөлдік басқару ақпараттық қауіпсіздіктің маңызды бөлігі болып табылады.

Әсет:
Рөлдік басқару backend деңгейінде іске асырылды. Users кестесінде role өрісі бар, ол user немесе admin болуы мүмкін. Admin маршруттарына сұраныс келген кезде сервер алдымен JWT токенді тексереді, содан кейін деректер базасынан қолданушының рөлін қарайды.

Егер қолданушы admin болмаса, басқару әрекеттеріне рұқсат берілмейді. Бұл жерде тағы да негізгі тексеру frontend-та емес, backend-та жасалады. Себебі қауіпсіз жүйеде рұқсат логикасы міндетті түрде сервер жағында болуы керек.

## 15-слайд. Әрекет тарихы

Жанагүл:
Бұл слайдта әрекеттер тарихы көрсетілген. Жүйеде маңызды әрекеттер журналға жазылады: тіркелу, кіру, шығу, құжат жүктеу, құжатты қарау, бөлісу, өшіру, 2FA қосу немесе өшіру, admin әрекеттері. Бұл функция жүйедегі белсенділікті бақылау үшін керек.

Қолданушы өз аккаунтында қандай әрекеттер болғанын көре алады. Ал әкімші жалпы жүйедегі маңызды оқиғаларды талдай алады. Бұл ақпарат күмәнді белсенділікті байқауға көмектеседі.

Әсет:
Activity logs қауіпсіздік аудитінің қарапайым түрі болып есептеледі. Егер бір аккаунтта күмәнді әрекет болса, мысалы белгісіз құрылғыдан кіру, құжатты жиі ашу немесе бөлісу, оны журнал арқылы көруге болады.

Логта action_type, action_details, created_at сияқты мәліметтер сақталады. Кейбір жағдайда IP және user-agent сияқты техникалық ақпарат та қолданылады. Бірақ логта құжаттың нақты мазмұны сақталмайды, себебі ол құпиялылыққа әсер етуі мүмкін. Біз тек әрекет фактісін тіркейміз.

## 16-слайд. Қолданыстағы жүйелермен салыстыру

Әсет:
Бұл слайдта AuthGuard Locker қолданыстағы файл сақтау жүйелерімен салыстырылады. Мысалы, Google Drive немесе Yandex Disk сияқты сервистер файл сақтау және бөлісу үшін өте ыңғайлы. Бірақ біздің жобаның мақсаты олармен толық коммерциялық деңгейде бәсекелесу емес.

Біздің ерекшелігіміз - ақпараттық қауіпсіздік механизмдерін бір оқу жобасының ішінде практикалық түрде көрсету. Жүйеде email растау, JWT авторизация, bcrypt хэштеу, TOTP 2FA, AES-256-GCM файл шифрлауы, уақытша сілтеме, рөлдік басқару, admin panel және activity logs бар. Сонымен қатар криптографиялық модуль арқылы теориялық алгоритмдерді де көрсетуге болады.

Жанагүл:
Яғни AuthGuard Locker ұйым ішіндегі құжаттарды қауіпсіз сақтау және оқу-тәжірибелік мақсатта қауіпсіздік принциптерін көрсету үшін жасалды. Жүйе шағын ұйымға, оқу орнына немесе ішкі құжат айналымына бейімделе алады.

Салыстыру арқылы біз жобаның негізгі құндылығын көрсетеміз: дайын бұлттық сервистер көп функция береді, ал біздің жүйе қауіпсіздіктің қалай іске асатынын толық түсіндіруге мүмкіндік береді.

## 17-слайд. Қолдану мүмкіндіктері

Жанагүл:
Бұл слайдта жүйені қай жерде қолдануға болатыны көрсетілген. Бірінші қолдану саласы - ұйым ішіндегі құжат айналымы. Мысалы, HR бөлімінде қызметкерлердің жеке құжаттары, еңбек келісімшарттары, анықтамалары және өтініштері сақталуы мүмкін. Мұндай құжаттарда жеке деректер болғандықтан, оларды ашық сақтау қауіпті.

Екінші қолдану саласы - компания ішіндегі келісімшарттар мен ішкі файлдарды басқару. Қолданушы құжатты жүктейді, қажет адамға жібереді немесе уақытша сілтеме береді. Сілтеменің мерзімі біткеннен кейін құжат автоматты түрде қолжетімсіз болады.

Әсет:
Үшінші қолдану саласы - білім беру ұйымдары. Мысалы, студенттердің өтініштері, дипломдық материалдар, ішкі бұйрықтар немесе оқытушыларға арналған құжаттар сақталуы мүмкін. Жүйені шағын ұйымға бейімдеп қолдануға болады.

Болашақта жүйені кеңейтуге болады: электрондық қолтаңба қосу, cloud object storage қолдану, толық аудит модулін дамыту, резервтік көшіру енгізу, корпоративтік LDAP немесе Active Directory арқылы кіруді қосу. Яғни жоба әрі қарай дамытылатын архитектурамен жасалды.

## 18-слайд. Тестілеу нәтижелері

Әсет:
Бұл слайдта тестілеу нәтижелері көрсетілген. Тестілеу кезінде алдымен негізгі функционал тексерілді: тіркелу, email-ға код жіберу, кодты растау, пароль енгізіп аккаунт жасау, жүйеге кіру, 2FA қосу, құжат жүктеу, құжатты қарау, уақытша сілтеме жасау және басқа қолданушыға жіберу.

Одан бөлек admin panel, рөл өзгерту, 2FA reset, қолданушыларды басқару және әрекеттер тарихы тексерілді. Backend API мен frontend беттері бірге жұмыс істейтіні қаралды.

Қауіпсіздік жағынан да бірнеше сценарий тексерілді: қате парольмен кіру, жарамсыз JWT токенімен қорғалған маршрутқа кіру, басқа қолданушының құжатын ашуға тырысу, мерзімі өткен уақытша сілтемені ашу, 2FA кодын қате енгізу. Мұндай сценарийлерде жүйе рұқсат бермеуі керек, және тестілеу барысында осы логика тексерілді.

Жанагүл:
Нәтижесінде қорғалған веб-қосымша жұмыс істейтін деңгейге жеткізілді. Пайдаланушы интерфейсі ашылады, backend API сұраныстарды қабылдайды, PostgreSQL деректер базасы қосылған, email арқылы код жіберіледі, құжаттар жүктеледі және шифрланған түрде сақталады.

Deploy жағынан frontend Vercel платформасына, backend Render платформасына орналастырылды. Бұл жобаны тек локалды компьютерде емес, интернет арқылы да көрсетуге мүмкіндік береді.

## 19-слайд. Қорытынды

Жанагүл:
Дипломдық жоба барысында AuthGuard Locker атты қорғалған веб-қосымша әзірленді. Жүйе құжаттарды қауіпсіз сақтау, басқару және бөлісуге арналған. Жоба барысында пайдаланушыға ыңғайлы интерфейс жасалды, тіркелу, кіру, email растау, dashboard, құжат жүктеу, құжатты қарау және бөлісу функциялары іске асырылды.

Жобаның негізгі нәтижесі - қолданушы құжатты жүйеге жүктеп, оны қорғалған түрде сақтай алады және қажет кезде бақыланатын түрде басқа адаммен бөлісе алады.

Әсет:
Техникалық жағынан жобаның негізгі бағдарламалау тілі JavaScript болды. Frontend-та React және Vite, backend-та Node.js және Express.js қолданылды, ал деректер PostgreSQL базасында сақталды. Қауіпсіздік механизмдері ретінде JWT авторизациясы, bcrypt пароль хэші, TOTP негізіндегі 2FA, AES-256-GCM файл шифрлауы, рөлдік басқару және activity logs енгізілді.

Осылайша жоба ақпараттық қауіпсіздіктің бірнеше маңызды механизмін бір жүйеге біріктіреді. Бұл дипломдық жұмыстың мақсаты орындалды деп айтуға болады: қазіргі аутентификация және шифрлау әдістерін қолдана отырып қорғалған веб-қосымша әзірленді.

## 20-слайд. Назар аударғаныңыз үшін рахмет

Жанагүл:
Құрметті комиссия мүшелері, біздің дипломдық жобамызды тыңдағандарыңызға рахмет. Біз AuthGuard Locker жүйесі арқылы құжаттарды қауіпсіз сақтау, пайдаланушыны сенімді тексеру және қолжетімділікті басқару мәселесін практикалық түрде шешуге тырыстық.

Әсет:
Енді жүйенің жұмысын қысқаша демонстрациялап көрсетуге дайынбыз. Сұрақтарыңыз болса, жауап беруге дайынбыз.

## Демо кезінде айтылатын қысқа сценарий

1. Басты беттен тіркелу формасын ашамыз.
2. Email енгізіп, растау кодын жібереміз.
3. Email-ға келген кодты енгізіп, парольді өзіміз жазамыз.
4. Жүйеге кіріп, dashboard бетін көрсетеміз.
5. 2FA баптау бетінде QR-код арқылы қосу принципін түсіндіреміз.
6. Құжат жүктейміз және оны "Менің құжаттарым" бөлімінде көрсетеміз.
7. Құжаттың қауіпсіз сақталатынын, backend-та AES-256-GCM қолданылатынын айтамыз.
8. Уақытша сілтеме жасап, мерзімі бар екенін көрсетеміз.
9. Admin panel-де қолданушы рөлдерін басқару мүмкіндігін көрсетеміз.
10. Әрекет тарихында логтар жазылатынын көрсетеміз.

## Комиссия сұрақтары және дайын жауаптар

1. Жобаңыздың негізгі мақсаты қандай?
Жауап: Құжаттарды қауіпсіз сақтау, басқару және бөлісуге арналған қорғалған веб-қосымша әзірлеу.

2. Неге бұл тақырып өзекті?
Жауап: Құжат айналымы цифрлық форматқа көшкен сайын жеке деректердің таралу қаупі өседі, сондықтан аутентификация, авторизация және шифрлау қажет.

3. Жүйенің атауы қандай?
Жауап: AuthGuard Locker.

4. Frontend қандай технологиямен жасалды?
Жауап: React және Vite арқылы жасалды.

5. Backend қандай технологиямен жасалды?
Жауап: Node.js және Express.js арқылы жасалды.

5.1. Жоба қай бағдарламалау тілінде жазылды?
Жауап: Жобаның негізгі бағдарламалау тілі - JavaScript. Frontend React арқылы, backend Node.js арқылы JavaScript экожүйесінде жасалды.

6. Деректер базасы қандай?
Жауап: PostgreSQL қолданылды, код SQL Server-ге де бейімделген.

7. Неге React таңдадыңыз?
Жауап: React компоненттік архитектура береді, интерфейсті бөліктерге бөліп жасауға ыңғайлы және SPA құруға қолайлы.

8. Неге Vite қолданылды?
Жауап: Vite development кезінде жылдам hot reload береді және production build-ті тез жасайды.

9. Неге Node.js таңдадыңыз?
Жауап: Node.js асинхронды I/O моделі арқылы API сұраныстарын тиімді өңдейді және frontend-пен бір тілде, JavaScript-та жұмыс істеуге мүмкіндік береді.

10. Express.js не үшін керек?
Жауап: REST API маршруттарын, middleware-ді және сұраныс-жауап логикасын жеңіл ұйымдастыру үшін.

11. JWT деген не?
Жауап: JWT - сервер қол қойған токен. Ол қолданушының кім екенін және қандай рөлі бар екенін API сұраныстарында тексеруге мүмкіндік береді.

12. JWT қандай бөліктерден тұрады?
Жауап: Header, Payload және Signature.

13. JWT қайда қолданылады?
Жауап: Қорғалған API маршруттарына кіргенде Authorization header арқылы жіберіледі.

14. Access token мен refresh token айырмашылығы қандай?
Жауап: Access token қысқа мерзімге API-ға кіруге арналған, refresh token жаңа access token алу үшін ұзақтау сақталады.

15. Парольдер қалай сақталады?
Жауап: Ашық түрде сақталмайды, bcryptjs арқылы хэштеледі.

16. Неге парольді шифрламай, хэштейміз?
Жауап: Хэш біржақты функция. Парольді қайта ашу мүмкін болмауы керек, тек енгізілген парольдің хэші салыстырылады.

17. bcrypt артықшылығы қандай?
Жауап: Ол баяу хэш функциясы, salt қолданады және brute force шабуылын қиындатады.

18. 2FA деген не?
Жауап: Екі факторлы аутентификация. Парольден бөлек қосымша бір реттік код талап етеді.

19. TOTP деген не?
Жауап: Time-based One-Time Password, уақытқа негізделген бір реттік код.

20. 2FA қалай іске асырылды?
Жауап: speakeasy арқылы құпия кілт жасалады, qrcode арқылы QR көрсетіледі, қолданушы authenticator app-пен код алады.

21. 2FA коды қанша таңбадан тұрады?
Жауап: Әдетте 6 таңбадан тұрады.

22. Егер қолданушы 2FA телефонын жоғалтса ше?
Жауап: Admin panel арқылы 2FA reset жасау функциясы қарастырылған.

23. AES-256-GCM не үшін қолданылды?
Жауап: Құжаттарды серверде шифрланған күйде сақтау үшін.

24. AES симметриялық па, асимметриялық па?
Жауап: Симметриялық алгоритм, бір құпия кілтпен шифрлау және дешифрлау жасалады.

25. GCM режимінің артықшылығы қандай?
Жауап: Ол деректі шифрлап қана қоймай, authentication tag арқылы тұтастығын тексереді.

26. Файл қашан шифрланады?
Жауап: Серверге жүктелгеннен кейін, сақталмай тұрып шифрланады.

27. Файл қашан дешифрланады?
Жауап: Қолданушы preview немесе download жасағанда, рұқсат тексерілгеннен кейін.

28. Құжатқа басқа қолданушы қол жеткізе ала ма?
Жауап: Тек иесі, жіберілген алушы немесе жарамды уақытша сілтемесі бар адам ғана.

29. IDOR шабуылынан қалай қорғандыңыз?
Жауап: Backend әр сұраныста құжат иесін немесе рұқсатты тексереді.

30. Уақытша сілтеме қалай жұмыс істейді?
Жауап: Сервер token жасайды, оны shared_links кестесінде сақтайды және expires_at арқылы мерзімін тексереді.

31. Сілтеме мерзімі өтсе не болады?
Жауап: Сервер құжатты көрсетпейді, expired жауап қайтарады.

32. Рөлдік басқару қалай жасалған?
Жауап: Users кестесінде role өрісі бар. Admin маршруттарына тек role=admin қолданушы кіре алады.

33. Admin panel не істейді?
Жауап: Қолданушыларды көру, қосу, өшіру, рөл өзгерту, 2FA reset жасау.

34. Activity log не үшін керек?
Жауап: Жүйедегі маңызды әрекеттерді бақылау және қауіпсіздік оқиғаларын талдау үшін.

35. Логта қандай деректер сақталады?
Жауап: action_type, action_details, created_at, кей жерде IP және user-agent.

36. Қолданушы әрекеттерін жазу құпиялылыққа әсер етпей ме?
Жауап: Логта құжат мазмұны сақталмайды, тек әрекет туралы қысқа мәлімет сақталады.

37. Email коды не үшін керек?
Жауап: Тіркеліп жатқан email шынымен қолданушыға тиесілі екенін растау үшін.

38. Email жіберу қандай сервис арқылы істейді?
Жауап: Mailjet API арқылы, қажет болса Brevo немесе Resend fallback ретінде қолдануға болады.

39. Email келмесе не болады?
Жауап: Жүйе кодты экранда көрсетпейді, email сервисін тексеру керек деген қате береді.

40. Неге кодты экранда көрсетпейсіз?
Жауап: Қауіпсіздік үшін. Код тек email иесіне баруы керек.

41. Frontend пен backend қалай байланысады?
Жауап: Axios арқылы REST API сұраныстары жіберіледі.

42. CORS не үшін керек?
Жауап: Frontend доменінен backend API-ға рұқсатты басқару үшін.

43. Helmet не үшін қолданылады?
Жауап: HTTP security headers қосу үшін.

44. Rate limiting не үшін қажет?
Жауап: Бір IP-ден тым көп сұранысты шектеу, brute force шабуылын азайту үшін.

45. Multer не үшін қолданылды?
Жауап: Файл жүктеу, multipart/form-data сұраныстарын өңдеу үшін.

46. Файл көлеміне шектеу бар ма?
Жауап: Иә, backend-та maximum upload size конфигурацияланады.

47. Қандай файл түрлері қолдайды?
Жауап: PDF, Word, Excel, PowerPoint, суреттер, мәтіндік файлдар.

48. PowerPoint preview қалай жасалады?
Жауап: Мүмкін болса LibreOffice арқылы PDF preview, болмаса PPTX ішіндегі мәтін HTML preview ретінде көрсетіледі.

49. Неге LibreOffice керек?
Жауап: Office файлдарын серверде PDF-ке конвертациялап, браузерде көрсету үшін.

50. Құжаттарды database-та сақтау дұрыс па?
Жауап: Кіші және орта файлдар үшін мүмкін. Үлкен production жүйеде object storage қолданған дұрыс.

51. Production үшін не жақсартар едіңіз?
Жауап: Object storage, HttpOnly cookie, толық CSP, backup, monitoring, audit export қосар едік.

52. LocalStorage-та JWT сақтау қауіпсіз бе?
Жауап: Ыңғайлы, бірақ XSS болса қауіп бар. Production-та HttpOnly cookie жақсырақ.

53. XSS-тен қалай қорғанасыз?
Жауап: React әдепкі escaping жасайды, қауіпті HTML қолданбауға тырыстық, input validation бар.

54. SQL injection қалай алдын алынды?
Жауап: Parameterized query және prepared input қолданылды.

55. CSRF қаупі бар ма?
Жауап: JWT Authorization header арқылы жіберілсе CSRF қаупі cookie-based auth-қа қарағанда төмен.

56. Құпия кілттер қайда сақталады?
Жауап: Environment variables ішінде, кодқа commit жасалмайды.

57. ENCRYPTION_KEY не үшін керек?
Жауап: AES-256-GCM шифрлау кілтін алу үшін.

58. JWT_SECRET не үшін керек?
Жауап: JWT токеніне қол қою және оны тексеру үшін.

59. PostgreSQL-де қандай негізгі кестелер бар?
Жауап: users, documents, shared_links, activity_logs, refresh_tokens, token_blacklist.

60. Users кестесінде не бар?
Жауап: email, password_hash, role, is_verified, verification_code, twofa_secret, twofa_enabled.

61. Documents кестесінде не бар?
Жауап: title, category, filename, mime_type, file_size, file_data, user_id, deleted_at.

62. Shared_links кестесінде не бар?
Жауап: document_id, token, expires_at, created_by.

63. Refresh token не үшін blacklist/revoke болады?
Жауап: Logout немесе token rotation кезінде ескі token қолданылмауы үшін.

64. Token blacklist не үшін керек?
Жауап: Access token мерзімі бітпей тұрып logout жасалғанда оны жарамсыз ету үшін.

65. Басты қауіпсіздік қағидаңыз қандай?
Жауап: Defense in depth, яғни бірнеше қорғаныс қабатын қолдану.

66. Жүйеде қандай қолданушы рөлдері бар?
Жауап: user және admin.

67. Admin өзін өшіре ала ма?
Жауап: Интерфейсте өзін өшіру немесе admin рөлін алып тастау шектелген.

68. Құжатты өшіргенде бірден жойыла ма?
Жауап: Жүйеде deleted_at арқылы soft delete логикасы бар, қажет болса қалпына келтіру сценарийі қолданылады.

69. Себет функциясы бар ма?
Жауап: Иә, өшірілген құжаттарды бөлек көрсету және басқару қарастырылған.

70. Құжатты басқа user-ге жіберу қалай жұмыс істейді?
Жауап: Алушы email бойынша users кестесінен табылады, document_transfers арқылы байланыс жасалады.

71. Алушы тіркелмеген болса ше?
Жауап: Жүйе "тіркелген қолданушы табылмады" деп қате қайтарады.

72. Неге тіркелмеген email-ге құжат жібермейсіз?
Жауап: Құжатқа қолжетімділікті жүйе ішінде бақылау үшін.

73. Уақытша сілтемені кез келген адам аша ала ма?
Жауап: Иә, token бар адам аша алады, бірақ мерзімі шектеулі.

74. Уақытша сілтеменің қауіпі қандай?
Жауап: Token басқа адамға түссе ашуы мүмкін, сондықтан мерзім шектеуі және күрделі random token қажет.

75. Token қалай жасалады?
Жауап: Crypto random мәндері арқылы болжанбайтын token жасалады.

76. Құжат preview қауіпсіз бе?
Жауап: Preview алдында рұқсат тексеріледі және файл тек сұраған кезде дешифрланады.

77. PDF пен DOCX preview айырмашылығы?
Жауап: PDF браузерде тікелей ашылады, DOCX/Office файлдары конвертация немесе HTML preview арқылы көрсетіледі.

78. Неге криптографиялық модуль қостыңыз?
Жауап: Шифрлау, хэштеу, RSA принциптерін тәжірибе жүзінде көрсету үшін.

79. SHA-256 не үшін керек?
Жауап: Деректен тұрақты хэш алу, тұтастықты тексеру және криптографияны түсіндіру үшін.

80. RSA не үшін көрсетілген?
Жауап: Асимметриялық криптография принципін көрсету үшін.

81. AES пен RSA айырмашылығы қандай?
Жауап: AES симметриялық, бір кілт; RSA асимметриялық, public/private key жұбы.

82. Жобаның ғылыми жаңалығы неде?
Жауап: Бір оқу жобасында аутентификация, 2FA, JWT, AES-GCM, secure sharing және audit log механизмдері біріктірілді.

83. Практикалық маңызы қандай?
Жауап: HR, компания ішкі құжат айналымы, білім беру ұйымдарында құжаттарды қауіпсіз сақтау үшін қолдануға болады.

84. Google Drive-тен айырмашылығы қандай?
Жауап: Біздің жүйе ұйым ішіндегі қауіпсіздік механизмдерін ашық көрсетеді: рөлдік басқару, 2FA reset, криптомодуль, audit log.

85. Жүйе коммерциялық дайын өнім бе?
Жауап: Бұл дипломдық практикалық прототип, production үшін қосымша backup, monitoring және storage жақсарту керек.

86. Қандай тесттер жүргізілді?
Жауап: Функционалдық тесттер: тіркелу, кіру, құжат жүктеу, бөлісу, 2FA, admin panel. Қауіпсіздік тесттері: қате token, рұқсатсыз құжат, мерзімі өткен link.

87. Қандай қателер кездесті?
Жауап: Deployment кезінде database URL, email service, LibreOffice preview сияқты мәселелер болды, олар конфигурация және fallback логикасы арқылы түзетілді.

88. Render не үшін қолданылды?
Жауап: Backend-ті public API ретінде орналастыру үшін.

89. Vercel не үшін қолданылды?
Жауап: Frontend React қосымшасын орналастыру үшін.

90. Backend URL frontend-ке қалай беріледі?
Жауап: VITE_API_BASE_URL environment variable арқылы.

91. DATABASE_URL не үшін керек?
Жауап: Backend PostgreSQL деректер базасына қосылуы үшін.

92. SSL database connection керек пе?
Жауап: Cloud PostgreSQL қолданғанда PGSSLMODE=require арқылы SSL қосылады.

93. Жүйеде localization бар ма?
Жауап: Иә, қазақша, орысша, ағылшынша тіл ауыстыру қарастырылған.

94. Неге интерфейс қазақша?
Жауап: Қорғау және мақсатты қолданушы аудиториясы үшін қазақша негізгі тіл ретінде таңдалды.

95. Неге email код 10 минут жарамды?
Жауап: Қауіпсіздік пен ыңғайлылық арасындағы баланс. Ұзақ болса қауіп, өте қысқа болса қолданушыға ыңғайсыз.

96. Пароль саясаты қандай?
Жауап: Кемінде 8 таңба, бас әріп, кіші әріп және сан талап етіледі.

97. Қолданушы парольді ұмытса ше?
Жауап: Forgot password арқылы email-ға reset code жіберіліп, жаңа пароль қойылады.

98. Email service істемесе не болады?
Жауап: Код экранда көрсетілмейді, жүйе email сервисін тексеру қажет екенін хабарлайды.

99. Жобаны болашақта қалай дамытасыз?
Жауап: HttpOnly cookie, object storage, CSP, full audit export, backup, notifications, электрондық қолтаңба қосуға болады.

100. Жобаның қорытынды нәтижесі қандай?
Жауап: AuthGuard Locker атты қорғалған веб-қосымша әзірленді, онда тіркелу, кіру, 2FA, құжаттарды шифрлау, қауіпсіз бөлісу, admin panel және әрекеттер журналы іске асырылды.

## Ең жиі қойылатын 10 сұраққа өте қысқа жауап

1. Неге AES-256-GCM? - Құпиялылық пен тұтастықты бірге береді.
2. Неге bcrypt? - Парольді ашық сақтамайды және brute force-ты қиындатады.
3. Неге JWT? - Stateless API authorization жасауға ыңғайлы.
4. Неге 2FA? - Пароль ұрланса да қосымша қорғаныс береді.
5. Неге React? - Компоненттік, жылдам, SPA жасауға ыңғайлы.
6. Неге Node/Express? - REST API тез құрылады, JavaScript ecosystem кең.
7. Неге PostgreSQL? - Сенімді relational database, cloud-та оңай deploy.
8. Құжат қалай қорғалады? - Рұқсат тексеріледі, файл AES-GCM арқылы шифрланады.
9. Admin не істейді? - Қолданушы, рөл, 2FA және жүйе күйін басқарады.
10. Болашақ жақсарту? - HttpOnly cookie, cloud object storage, backup, CSP, monitoring.
